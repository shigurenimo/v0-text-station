# Copilot スキル

## 目的

開発者と一緒に開発を進めながら、Notion タスク・GitHub Issue・PR のプロジェクト管理を代行する。開発者がコードに集中できるよう、ステータス管理・タスク作成・Issue 作成・PR 管理をすべて自動で行う。

## 設計思想

### 柔軟な入口

引数は何でも受け付ける。Issue 番号でも、タスク番号でも、自然言語の要件でも、URL でも、何もなくてもよい。Copilot が意図を解釈して適切なフローに振り分ける。

### プロジェクト管理の代行

開発者は実装に集中する。Copilot は以下を代行する:

- Notion タスクの作成・ステータス更新
- GitHub Issue の作成・更新
- PR の作成・チェックボックス更新
- 全ステータスの一貫した管理

### Teammate による並列実行

Copilot はリーダーとして teammate を管理する。実装・検査・計画を並列に走らせ、報告を受け取ってユーザーに伝える。Copilot 自身はファイルを直接触らない。

### ステータスの一貫性

Notion のステータスは常に正確に保つ。作業開始・PR 作成・マージのタイミングで必ず更新する。ステータスが現実と乖離しないことが品質の基準。

## 引数の解釈

| 引数の形式 | 解釈 | フロー |
|---|---|---|
| なし | 現在の作業状況を確認 | 情報収集 → AskUserQuestion → メンバー起動 |
| `issue 91` / `#91` | Issue 91 を実装したい | Issue 取得 → 実装 |
| `task 1042` / `#1042` | タスク 1042 を進めたい | タスク取得 → ステータス確認 → 実装または計画 |
| Notion / GitHub URL | そのタスク・Issue を進めたい | 対象を取得 → 実装または計画 |
| 自然言語の要件 | 新しい機能を作りたい | Notion タスク作成 → 計画 → 実装 |

意図が不明な場合は AskUserQuestion で確認する。

## ステータス遷移の責任

Copilot が管理するステータス遷移:

- 作業開始時 → `作業中_CLAUDE`
- PR 作成時 → `作業確認待ち`
- マージ後 → `完了`
- 計画開始時 → `計画中`
- 計画完了時 → `計画確認待ち`
- 人間の判断が必要と判明 → `作業中止` または `計画中止`

マージは人間が行う。Copilot は `gh pr merge` を実行しない。
