# /docs features-sync

コードから機能を吸い出し、`.docs/features/` を最新に保つ。粒度ルールに従ってコードと .md を 1:1 で整合させる。

## 起動条件

`/docs features-sync` 単独、または `/docs features-sync {対象}` で起動する。

- `/docs features-sync` — 全 routes と全 features を突合
- `/docs features-sync {dir}` — 1 機能ディレクトリだけ（例: `00010000.owner-dashboard.*`）
- `/docs features-sync --apply` — プレビュー表示せず即適用（デフォルトはプレビュー）

## コスト

中〜大。dir 数 × 1 agent。コードベースの規模に応じて 20〜100 agent。budget.total を見てスケールする。

## 機能粒度の判定ルール

「機能単位で検品・仕様詰めできるか」を基準に判定する。

### 機能として扱う

- フォーム（1 フォーム＝1 機能、Input 単体は数えない）
- フィルター・検索（条件絞り込みのロジックがある）
- KPI カード（計算式・データソースごとに別機能）
- チャート・テーブル（表示する集計式 / 列構成ごと）
- ダイアログ・モーダル（独立したインタラクション）
- 印刷・出力（成果物単位）
- 通知・アラート（出現条件付き）

### 機能として扱わない

- 個別 Input / Select / Button（フォームに吸収）
- 作成・削除・保存・キャンセル アクション（CRUD は親フォーム / カードに吸収）
- ページ遷移リンク・「〜に戻る」「詳細を見る」
- ヘッダー・タイトル・ラベル・見出し
- 空状態 / 準備中 / 注記 / 凡例
- バッジ単体・色分け（親カードに吸収）
- 印刷物のセクション見出し（レイアウト要素）
- 装飾（背景あしらい・OrbBlur 等）
- ページネーション（一覧画面の当然装備、機能としてカウントしない）

### 計算式・データソースが違うものは別機能

同じ grid に並ぶ KPI でも、計算式・データソースが違えば別 feature。

- 入居率 / 利用率 / 職員数 / 売上 → 個別 4 機能
- 月平均 / 前年比 / 当月実績 → 同じ指標の別断面なら親に吸収するが、別計算で別カードなら別機能

装飾的グループ（grid・余白）だけが理由で 1 つに括らない。理由: 計算式の spec 突合・データソース確認・閾値判定をそれぞれ独立に詰めるため。

## マーキング方式

`data-feature-id="<8桁ID>"` を JSX 要素に付ける。`<KpiCard>` のような共通プリミティブは `featureId` プロパティ経由で受け取って `data-feature-id` を出す実装にする。

8 桁 ID 体系。

- 前 4 桁 = 親機能（ディレクトリ）
- 後 4 桁 = 子トピック（0001 以降の連番、欠番は詰めない）

## 出力

### プレビューモード（デフォルト）

dir ごとに次を出す。

- **追加候補**: コードに wiring 候補があるが `.md` 無し
- **削除候補**: `.md` あるが JSX に wiring 無し（Phase 2 機能は除外して報告）
- **粒度違反**: 上記ルールに反する .md（フォーム入力・遷移ボタン・色分けバッジなど）
- **重複**: 同じ JSX 要素を指す複数 ID、または近似名称の `.md`
- **mismatch**: `.md` 本文と JSX 挙動の乖離

### --apply モード

候補を自動適用する。

- 追加: `.md` 新規作成 + JSX に `data-feature-id` / `featureId` 付与
- 削除: `.md` 削除 + JSX から属性除去
- 統合: 残す親 ID に統合先 JSX 要素を集約
- mismatch: `.md` 本文を JSX 挙動に合わせて書き直す

適用後に `feature-docs-manifest.json` を再生成（プロジェクトに manifest スクリプトがある場合）。typecheck を回して緑になったらコミット。

## 実行フロー

### Step 1 インベントリ

- 全 route / page コンポーネントを列挙
- 既存 `.md` の id・name・本文を読み出し
- コードに付いている `data-feature-id` / `featureId` 値を grep で全件回収

### Step 2 並列レビュー

dir ごとに 1 agent を立て、上記ルールで次を判定。

- JSX 要素 ↔ `.md` の 1:1 対応
- 粒度違反の有無
- 計算式が違う KPI が一括 ID になっていないか
- `.md` 本文と実装の乖離

### Step 3 統合

dir ごとの判定をマージし、追加・削除・統合・mismatch のアクションリストを作成。

### Step 4 適用（`--apply` のみ）

`hiract-developer` などの実装エージェントに渡してアクションを実行。`.md` ファイル操作と JSX 属性編集を 1 トランザクションで進める。

### Step 5 検証

- typecheck（プロジェクトの `bun run typecheck` 等）
- manifest 再生成
- code only ID 数 = 0、orphan は Phase 2 機能のみ、を確認

## 注意

- 親 `.md`（`NNNN0000.*.md`）は触らない（粒度監査の対象外）
- `stories/` は触らない
- 共通プリミティブ（`ui/card.tsx` `ui/button.tsx` 等）には `data-feature-id` を付けない（呼び出し側で付ける）
- 既存 ID を再利用しない（削除した番号は欠番のまま残す）
- Phase 2 機能で `.md` だけ先行しているものは `--apply` 時も削除しない（status・notes で識別）
