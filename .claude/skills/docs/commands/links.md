# /docs links

`.docs/` 配下の `[[wikilink]]` を全件解決し、未定義リンク・孤児ページ・循環参照を検出する。グラフとして健全に保つための健康診断。

## 起動条件

`/docs links` 単独で起動する。

## コスト

軽い。grep ベースの静的解析が主体。agent は最終整理に 1〜2 体。

## 出力

- 未定義リンク（`[[name]]` の name に対応するファイルが無い）→ tasks.md の「人間判断待ち」または相当セクションへ「次に書くべきノード」として追記
- 孤児ページ（どこからもリンクされていない .md）→ レポートに列挙。削除・index 追加・親ページからのリンク追加のいずれかを促す
- 循環参照（A→B→A の閉路で他とつながらないもの）→ レポートに列挙
- 表記揺れ（`[[A]]` と `[[a]]` が混在、エイリアスで吸収すべきもの）→ レポートに列挙

軽微なものは自動修正の選択肢を出す。判断が要るものだけ tasks.md へ流す。

## 実行フロー

### Step 1 リンクグラフ構築

`.docs/` 全 .md ファイルを走査し、各ファイルから `[[...]]` を抽出してグラフを組む。

- ノード：.md ファイル
- エッジ：`[[name]]` または `[[path|alias]]`
- name→ファイル解決：slug マッチ・ファイル名マッチ・frontmatter name マッチ

### Step 2 異常検出

- 未定義リンク：name が解決できない `[[...]]`
- 孤児ノード：入次数 0 のページ（index.md・README・トップから到達不能）
- 循環参照：他とつながらない閉路
- 表記揺れ：同じ意味の異なる name が複数存在
- frontmatter name と slug の不一致

### Step 3 振り分け

- 未定義リンク：tasks.md「人間判断待ち」へ「次に書くべきノード」として追記
- 孤児ノード：レポートに列挙し、削除・index 追加・親リンク追加の判断を促す
- 循環参照：レポートに列挙
- 表記揺れ：エイリアスへの寄せ方を 1 案提示

### Step 4 報告

- スキャンしたファイル数・抽出した `[[...]]` 数
- 未定義リンク件数・孤児件数・循環件数・表記揺れ件数
- tasks.md へ追記した件数
- 自動修正可能な件数と例

## 実装ガイド

- Workflow は不要（軽い処理）
- Bash の grep / find で静的解析を主体にする
- 解決ロジックは frontmatter の `name:` と slug を優先、その後ファイル名で fallback
- writing-rules.md の wikilink ルールに従う

## 禁止事項

- 未定義リンクを勝手に作成しない（次に書くべきノードのしるしとして残す）
- 孤児ページを勝手に削除しない（人間判断）
- 表記揺れを勝手に統一しない（候補だけ示す）
