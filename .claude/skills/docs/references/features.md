# features

機能単位で「何ができるか」を書く。実装詳細は書かない。コードから生成・更新する。

## 規模で2モード

小規模（〜15機能）。

- `features.md` 単一ファイルに全機能を並べる
- 1機能は見出し+箇条書き3〜8行

中〜大規模（15機能以上）。

- `features/{slug}.md` に1機能1ファイル
- `features/index.md` をロール別または責務別の目次にする

切り替えの判断は機能数だけでなく「索引が必要になったか」「機能ごとに背景や運用注記を持ちたくなったか」で行う。

## features.md（単一ファイル）

```md
# 機能一覧

## {機能名}

- {何ができるか1}
- {何ができるか2}
- ステータス: {状態の種類}

## {機能名}

- {何ができるか1}
- {何ができるか2}
```

ルール。

- 見出しは機能名（日本語）
- 箇条書きは「何ができるか」を書く。「どう実装されているか」は書かない
- 1機能あたり3〜8行
- 廃止された機能は書かない（backlogs/ に移すか signals/ に痕跡を残す）

## features/{slug}.md（分割）

各機能を1ファイルに切り出す。

frontmatter。

```yaml
---
aliases: [{機能名の別表記}]
id: F{NNN}
name: 日本語の機能名
slug: english-with-hyphens
milestone: YYYY-MM-DD
models: [{domain}, {domain}]
pages: [{route}, {route}]
stories: [{ストーリー名}]
---
```

フィールド。

- `name` / `slug`: 必須
- `aliases`: wikiリンクの別表記用（`[[エイリアス]]` で解決させる）
- `id`: 全機能で一意の ID（`F010` 形式）。番号プレフィックス規則を採るなら NNN と揃える
- `milestone`: リリース計画と連動させたいとき。`milestones.md` の日付と揃える（詳細は [milestones.md](milestones.md)）
- `models` / `pages` / `stories`: クロスリンク。それぞれ `models/{name}.md` / `pages/{route}.md` / `stories/{name}.md` の識別子を配列で持つ。features → 関連リソースの逆引きを支える
- 優先度フィールド `priority: must/nice` は使わない。優先度はマイルストーン日付だけで線を引く

本文。

```md
# {機能名}

{ユーザーが何をできるか・何が見えるかを自然な日本語で記述する}

{必要に応じて箇条書きで補足}

備考: {制約・未対応事項。あれば}
```

書き方。

- 顧客（その製品の利用者）が読んで「何のための機能か・どんな価値があるか・どう計算されるか」が分かる内容にする。仕様書ではなく価値説明として書く
- 「ユーザーが何をできるか／何が見えるか」を主語に書く。コンポーネント名・テーブル名・関数名・API パス・ファイルパスなど実装詳細は書かない
- UI デザイン・レイアウトは書かない。画面部品の名前（カード・バッジ・タブ・パネル・進捗バー・フォーム・モーダル・ダイアログ・グリッド・サイドバーなど）や配置・色・クリックなどの操作記述を本文に持ち込まない。「どんな画面か」ではなく「何のための機能か」を書く
- 計算式・仕組みは書く。金額の算定式・按分・KPI の定義・集計の考え方は、顧客が結果を信頼するために必要なので業務の言葉で残す（例「介護報酬額＝単位数合計×地域単価（円未満切り捨て）」「入居率＝入居者数÷定員」）。これは「書かない実装詳細」ではなく「書くべき仕組み」
- コードから抽出した実装仕様を別節（「実装仕様」「データ操作」など）にまとめて置かない。コードを読めば分かることは節ごと不要
- 業務上の数字や事実は具体に
- 専門用語・他機能名・業界用語は wikiリンク `[[]]` でつなぐ。節ごとに初出1回が目安。専門用語は初出時に短く説明する
- 1機能あたり5〜20行を目安に
- 廃止された機能は書かない

サブ機能（トピック）に切り出す粒度。

機能を細かいトピックに分けてファイル化するとき、1ファイルに値するのは「顧客に説明する価値があるもの」だけにする。次のものは独立ファイルにしない。親機能の本文に1文で吸収するか、書かない。

- 画面の操作・部品そのもの（絞り込み・並べ替え・月送り・ページ送り・タブ切替・印刷ボタン・新規追加ボタン・見出し/ヘッダー・戻る・空状態やローディングの表示・別画面への導線リンク）。これらは「どう操作するか」であって機能の価値ではない
- コードを読めば分かるだけの自明な挙動（一覧をキーワードで絞れる、対象月を選べる、など）

逆に、独立トピックに値するのは次のようなもの。計算式・按分・集計の仕組み、二重入力や誤登録を防ぐ仕組み、権限による操作の分け方、業務上の制約条件（有効期間・上限・締め後の書き換え禁止など）。判断基準は「画面の部品か、業務の仕組み・ルールか」。部品なら作らない、仕組み・ルールなら作る。

ファイル名の規則。

2つの命名規則のどちらかを採用する。プロジェクトで1つに統一し、混ぜない。

slug 規則（小規模・中規模向け）。

- ファイル名は `{slug}.md`（英字ハイフン区切り、frontmatter `slug` と一致）
- ファイルシステム上の並びは英字順になる
- 参照は `[[{slug}]]`
- 例: `monthly-pl.md`、`occupancy.md`、`budget.md`

番号プレフィックス規則（大規模向け）。

- ファイル名は `NNN_{日本語名}.md` または `NNN_{日本語名}/`（ディレクトリ化して中に資料を併置できる）
- ファイルシステム上の並びを明示制御できる。経営者向け 010-099、施設長向け 100-199、介護職員向け 200-299 のように予約レンジを切る
- frontmatter の `slug` は別立てで英字キーとして保持し、`features/index.md` と相互参照する
- 参照は `[[NNN_{日本語名}]]`（wikiリンクはファイル名で解決）
- 連番は途中削除があっても詰めない（欠番は履歴）
- 例: `010_経営者向けダッシュボード.md`、`020_月次損益分析.md`、`170_サービスシフト管理.md`

どちらの規則でも、frontmatter の `slug` は英字キーとして必須にしておく。`features/index.md` のエントリは `[[ファイル名|表示名]] ({slug}) — 説明` 形式で書き、ファイル名と slug を両方見せる。

ディレクトリ化（NNN_{日本語名}/）。

機能ごとに `stories/` や図版・補足資料を併置したい場合は、機能をディレクトリにする。

```
features/
  NNN_{日本語名}/
    {日本語名}.md         本体（frontmatter + 本文）
    stories/              この機能に紐づくストーリー
      {story 名}.md
    assets/               図版・スクショ（必要なら）
```

ディレクトリ化した場合の wikiリンク解決は `[[NNN_{日本語名}]]` で本体ファイルに飛ぶよう、本体ファイル名はディレクトリ名と揃える（または `index.md` を使う）。stories は機能直下に置くと、機能とユースケースの結びつきが構造上明示できる。横断的なストーリーは `stories/`（リポジトリ直下）に残し、機能固有のストーリーだけ features 配下に置く運用も可。

## features/index.md

ロール別または責務別の TOC。

```md
# 機能一覧

## {ロール名}

法人全体の経営判断を支える画面群。

- [[{slug}]] ({slug}) — {何をする画面かの1行サマリ}
- [[{slug}|{表示名}]] ({slug}) — {同上}
```

ルール。

- エントリは `[[ファイル名|表示名]] ({slug}) — 説明` 形式。ファイル名と表示名が同じなら `[[ファイル名]] ({slug}) — 説明`
- 括弧内は frontmatter の `slug` 値のみ。URL は説明文側に書く
- 分類は責務（機能の意味で属するロール）で行う。サイドバーの URL 配置とは別軸（[sitemap.md](sitemap.md) を参照）。同じ機能が両ファイルで違うセクションに登場することは設計通り
- ロール別 TOC でカバーされない機能は「全ロール共通」セクションを設け、組織横断の土台機能を集める

## user-flows.md

ユーザー種別ごとに主要導線を番号付きステップで書く。E2E テスト生成のソースになる。

```md
# ユーザーフロー

## {ユーザー種別}導線

### {導線名}

1. {ステップ}
2. {ステップ}
3. {ステップ} or {別の選択}
4. {ステップ}
```

ルール。

- ユーザー種別ごとにセクションを分ける
- 導線は「ユーザーが何をするか」で書く。「システムが何をするか」は書かない
- 番号付きステップ。箇条書きにしない
- 各ステップは1行。長い説明を入れない
- 分岐がある場合は「A or B」で書く
- 画面遷移が分かる粒度（ページ名や URL は不要）

## 実装ステータスの併記

`features.md`（または `features/{slug}.md`）の各機能行に、実装ステータスと決定ステータスを併記すると俯瞰が利く。

実装ステータス。

- 実装済: DB → API → UI が完全に連動し、実データで動く
- 一部: 主要動線は動くが端のフローが未完
- モック: UI はあるが表示はダミーデータ
- 未着手

決定ステータス。

- 決定: スコープと挙動が確定
- 要相談: 仕様の詰めが残っている

`features/index.md` で集計し、`milestones.md` の各日付セクションに「実装済 X、一部 Y、未着手 Z」のサマリを最終更新日付付きで載せる運用に向く。
