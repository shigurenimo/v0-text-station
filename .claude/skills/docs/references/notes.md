# notes/

長文の解説・調査メモ・一時的な棚卸し記録を置く場所。decisions（判断履歴）にも features（製品機能）にもならない、説明や作業ログ系のドキュメント。

## 役割

- 業界用語・制度・会計知識など、`references/terms/{用語}.md` の短い定義から文中でリンクする「深掘り」記事
- 顧客との会話や打ち合わせを受けた一時的な棚卸し・調査メモ（実装が片付いたら破棄してよい）
- 設計上の検討プロセス（最終結論は decisions/ に分離する）

## ファイル名

- 恒久的な解説: `{slug}.md`（例: `インシデントとヒヤリハット.md`、`サービスコードの仕組み.md`）
- 日付付きの一時メモ: `{slug}-YYYY-MM-DD.md`（例: `features-orphans-2026-06-22.md`）

## frontmatter

```yaml
---
title: 日本語のタイトル
date: YYYY-MM-DD       # 一時メモの場合のみ。恒久解説は省略可
tags:
  - notes
  - {テーマ}
ephemeral: true        # 用が済んだら削除して良いことを明示。恒久解説は false または省略
---
```

- `title`: 一覧表示やリンクで使う日本語タイトル。本文の H1 と一致させる
- `date`: 一時メモの起票日（YYYY-MM-DD 固定）。relative date（「先週」「今月」）は禁止
- `ephemeral`: `true` のメモは寿命があり、片付いたら削除する前提。`false` / 省略は恒久解説扱い

## 本文の書き方

- H1 はファイルの title と一致させる。日付付きメモは末尾に `（YYYY-MM-DD）` を入れて読み手にも一時性を伝える
- 一時メモには末尾に「次のアクション」と「破棄条件」を書く。何が起きたら捨てて良いかを明示する
- terms から文中でリンクする深掘り記事は、用語定義の繰り返しは書かず、背景・歴史・運用上の機微などに集中する

## 一時メモの典型パターン

`ephemeral: true` で書くケースの例。

- 大規模リファクタの仕掛り棚卸し（PR が全部マージされたら捨てる）
- 顧客ヒアリング直後の生情報（signals/ や decisions/ に整理し終えたら捨てる）
- データ移行・棚卸しの ToDo（移行完了で捨てる）
- 一過性のバグ調査メモ（fix がマージされたら捨てる）

## 削除タイミング

`ephemeral: true` のメモはレビュー時に削除条件を満たしているか確認する。残し続けると「古い情報が並んでいて何が現役か分からない」状態になる。判断に迷う場合は本文の「破棄条件」を見直して残す根拠を更新する。
