# page-capabilities.md

ロール × 画面で「できること」を網羅した詳細リスト。商談・社内説明・QA 時に該当画面のできることを引きたいときの一次情報。

`capabilities.md`（ロール別サマリ）が「営業の元ネタ」なのに対し、`page-capabilities.md` は「画面ごとの仕様レベルの詳細」。二層構造で運用する。

## いつ作るか

- 画面数が多く（30 以上）、ロールごとに見える画面・操作が異なる
- 商談やオンボーディングで「この画面で何ができるんですか」と聞かれる頻度が高い
- QA・カスタマーサポートが画面別の挙動を引きたい

画面数が少ないか、capabilities.md だけで足りるならば作らない。

## フォーマット

```md
# ロール別ページ別できること一覧

{製品名} で各ロール（…）が各ページで何ができるかを網羅した詳細リスト。商談・社内説明・QA 時の元情報。

サマリは [[capabilities]] を、機能単位の意図は [[features/index|features]] を参照。

ロールは次の N 種類。

- {role_key}（{日本語名}）: {1 行説明}
- ...

## {ロール名}（{role_key}）

### {URL} {画面名}

- {動詞で始まる 1 行}
- {動詞で始まる 1 行}

### {URL} {画面名}

- ...
```

## ルール

- H2 はロール、H3 は `{URL} {画面名}`
- H3 はそのロールでログインしたときのサイドバー表示順に並べる
- 行は動詞で始める 1 行
- 1 画面 5〜15 行を目安に
- 下層ページの機能は親ページにまとめる（重要度が高い下層は独立して扱う）
- 汎用 UI 操作（保存・キャンセル・戻る）と細かい入力項目（住所・電話）は省略・集約する
- フェーズ運用がある製品では、現フェーズで使えない画面は載せない（実装は維持しつつ載せないという判断もある）
- H2 / H3 に注記（要相談・Phase 2 など）を含められる

## capabilities.md との分担

- capabilities.md: ロール別の「何が得られるか」を 1 ロール 5〜10 行で
- page-capabilities.md: ロール別 × 画面別の「何ができるか」を画面ごと数行で

capabilities.md の各行は page-capabilities.md の複数行を集約したもの、と捉える。

## 維持の仕方

- features または pages が更新されたら、該当画面の節だけ追従
- 全体再生成はしない
- サイドバー順が変わったら H3 の並びも追従
